# Studio Run Generated Map Mod Visibility

## Why

Deploying files is not enough. Run in Game must prove that Civ7 has admitted
the generated Studio mod and can read its stable setup row before applying or
starting a game.

## Product Contract

- The generated mod has stable id `mod-swooper-studio-run` and stable script
  path `maps/studio-run.js`.
- Freshness belongs to generated and deployed bytes, digests, and request
  correlation, not to a request-specific filename.
- `@civ7/direct-control` owns the one-wire catalog, activation, setup-reload,
  and row-read operations.
- The package-owned control-oRPC single-player lifecycle composes those atoms
  once, after any saved-config load and before setup mutation.
- Catalog refresh is soft. Run in Game does not restart Civilization VII.

## Requires

- Request-correlated generation and deployment snapshots.
- The package-owned `lifecycle.singlePlayer.start` control-oRPC capability.
- Typed setup and runtime failure projection.

## Enables

- Saved-config reconciliation can reuse one admitted visibility owner without
  duplicating catalog refresh or row polling.
- Runtime attribution can bind one request to generated bytes, deployed bytes,
  the exact setup row, and the loaded game.

## Boundaries

Do not add Studio-local control scripts, a second visibility workflow, a
whole-process restart, request-specific script paths, or hand edits to generated
or deployed output. Public status remains redacted; exact paths and correlation
stay in explicit private diagnostics.

## Verification

Static tests prove the stable generated tree, unique action/criteria identity,
deployment digest equality, exact target-mod admission, and exact row matching.
A rendered Run in Game request proves the same stable row and correlation in a
live Civ7 process without changing its PID.

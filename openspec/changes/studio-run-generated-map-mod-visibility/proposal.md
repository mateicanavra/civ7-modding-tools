# Studio Run Generated Map Mod Visibility

## Why

The generated Studio-run mod is the runtime artifact that Civ7 must offer in
setup. The current failure shows deployed files can exist while setup sees only
source catalog rows such as `{swooper-maps}/maps/latest-juicy.js`, not the
request-local row under `{mod-swooper-studio-run}`. Before saved-config
composition is repaired, the generated mod itself must be treated as a
first-class map mod with enough metadata and deployed identity to become
visible to Civ7 setup.

## Consumer Impact

Run in Game diagnostics and live setup checks can distinguish "the generated
mod is not a visible map mod" from "the saved setup config removed the generated
mod from the active target." Users still see only safe public runtime category
and recovery copy.

## Stop Conditions

- Generated mod metadata does not include the request run artifact id and map
  row.
- Deployed snapshot digest does not match the generated mod source.
- Civ7 shell/setup row readback sees only source catalog rows.
- Duplicate or ambiguous action/row identity is found and not dispositioned.
- Public payloads expose generated file-system paths or private diagnostics.

## Behavior Verification

Behavior tests inspect generated `.modinfo`, config XML, localized text, map
script row, embedded correlation markers, and deployment snapshot digests for
the exact request artifact. Live checks read Civ7 setup/shell rows for the
generated `{mod-swooper-studio-run}/maps/${runArtifactId}.js` row before
saved-config composition is treated as green.

## Structural Enforcement

Permanent positive assertions:

- request-local generated map mods render the required file classes declared in
  `target-vocabulary.md`;
- deployment snapshots copy from the generated mod root for the same request;
- generated row identity is request-local and carries the run artifact id.

Use behavior tests for generated bytes and Habitat only for recurring ownership
or file-class topology. Do not duplicate renderer code in a pattern.

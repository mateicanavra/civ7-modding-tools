# Upstream feedback — design-sync / claude.ai-design token classifier over-captures Tailwind v4 internals

**To:** maintainers of the claude.ai/design design-system self-check (the
component that regenerates `_adherence.oxlintrc.json` / `_ds_manifest.json` and
backs `check_design_system`), and of the Claude Code `design-sync` skill.
**From:** MapGen Studio (`@swooper/mapgen-studio-ui`, DS project
`531d158d-a7f6-41cb-87a4-f0f8a5e521b0`), 2026-07-08.

## Summary

For any design system whose stylesheet is compiled Tailwind CSS v4 output, the
token scan classifies the framework's engine variables as authored design
tokens and reports two permanent, unactionable findings:

1. *"N of M tokens couldn't be classified as color/spacing/radius/shadow/font"*
   — the unclassifiable names are Tailwind's `@theme` defaults
   (`--animate-spin`, `--default-transition-duration`, `--tracking-*`, …) and
   `@property`-registered composition vars (`--tw-*`).
2. *"K custom properties declared under component-style selectors … move them
   under `:root`"* — these are `--tw-*` utility-state vars (e.g.
   `--tw-space-y-reverse` in `.space-y-* :where(& > :not(:last-child))` rules)
   plus the universal `@layer properties` reset. They are **required** to be
   selector-scoped; following the finding's advice (hoisting to `:root`)
   breaks `space-*`/`divide-*`/transform/gradient/ring/filter utilities.

In our project: 202 reported tokens = 136 unique names + scope-attached
duplicates; 78 are `@property`-registered `--tw-*`, ~12 are `@theme` defaults;
only ~46 are authored. The findings recur on every sync, cannot be fixed
project-side (synced artifacts are regenerated), and cannot be fixed repo-side
(the classifier is app-side; we verified the Claude Code binary and the staged
converter contain no token classifier and no annotation mechanism).

## Requested fixes

1. **Exclude framework-owned properties from the authored-token scan.** A name
   registered via an `@property` at-rule in the scanned stylesheet is
   definitionally engine plumbing. Additionally exclude the Tailwind v4
   internal namespace by prefix: `--tw-*`, `--default-*`, `--animate-*`,
   `--blur-*`, and the numeric scales (`--tracking-*`, `--leading-*`,
   `--text-*--line-height`). This clears both findings for every Tailwind v4
   design system at once.
2. **Recognize the shadcn-style HSL-triplet color pattern.** Values matching
   `^\d{1,3}(\.\d+)?\s+\d{1,3}(\.\d+)?%\s+\d{1,3}(\.\d+)?%$` (consumed as
   `hsl(var(--x))`) should classify as `color`. Today every semantic color
   token in a shadcn-derived system (`--background`, `--primary`,
   `--destructive`, …) lands as unclassifiable-by-value and is emitted as
   `"other"` in the `x-omelette.tokenKinds` map.
3. **Fix the `tokenKinds` name heuristic** (appears to substring-match kind
   keywords): `--tw-translate-y → "color"` (vs `--tw-translate-x →
   "spacing"`), `--tw-backdrop-blur → "color"`, `--tw-duration → "color"`,
   `--tw-gradient-position → "color"`. With fix 1 these disappear from the map
   entirely; with fix 2 the semantic colors classify correctly.
4. **Optional:** if a project-supplied classification input is ever supported
   (e.g. annotations or a token-metadata file the converter uploads), document
   it in the design-sync skill — the 2026-07 skill/converter has no such
   surface, which is why this report exists.

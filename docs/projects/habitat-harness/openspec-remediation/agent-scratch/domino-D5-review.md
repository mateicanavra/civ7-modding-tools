## Acceptance Verdict

Not accepted. D5 remains blocked.

The packet is a syntactically valid OpenSpec scaffold, but it is not yet an implementation-ready Baseline Authority contract. The generated scaffold collapses the D5 source packet's concrete state model into broad prose, leaves a D5/D8 ownership and sequencing ambiguity, and does not give validation gates precise enough to prove the baseline authority behavior. An implementation agent would still need to decide product/domain terms, baseline state variants, rule-introduction manifest shape, current-tree integrity semantics, and downstream consumer projection shape.

## P1 Findings

### P1-1: The OpenSpec spec delta does not encode the D5 baseline state model

The source D5 packet explicitly requires baseline states for explicit empty, explicit debt, external exception, malformed, missing, orphan, introduced-rule expansion, and shrink-only failure. The current code also exposes these as real concerns: `BaselineContractFailureReason` contains missing, malformed, orphan, external exception, comparison-base, base-registry, base-baseline, projection mismatch, parser-owned, and rule-introduction reasons at `/Users/mateicanavra/Documents/.nosync/DEV/worktrees/wt-codex-deep-habitat-openspec-remediation/tools/habitat-harness/src/lib/baseline.ts:24`, while `loadBaselineState`, `validateBaselineContract`, and `guardBaselineExpansion` implement separate state paths at `/Users/mateicanavra/Documents/.nosync/DEV/worktrees/wt-codex-deep-habitat-openspec-remediation/tools/habitat-harness/src/lib/baseline.ts:343`, `/Users/mateicanavra/Documents/.nosync/DEV/worktrees/wt-codex-deep-habitat-openspec-remediation/tools/habitat-harness/src/lib/baseline.ts:390`, and `/Users/mateicanavra/Documents/.nosync/DEV/worktrees/wt-codex-deep-habitat-openspec-remediation/tools/habitat-harness/src/lib/baseline.ts:423`.

The generated requirement only says baselines SHALL be explicit structural-debt authority and includes two scenarios: existing debt and new debt at `/Users/mateicanavra/Documents/.nosync/DEV/worktrees/wt-codex-deep-habitat-openspec-remediation/openspec/changes/deep-habitat-d5-baseline-authority/specs/habitat-harness/spec.md:3`. It omits required normative scenarios for malformed baseline JSON, non-array/non-string/duplicate/unsorted rows, missing baselines, orphan baselines, external exception projection equality, parser-owned baselining, comparison-source failures, introduced-rule manifest mismatch, and shrink-only mutation refusal. The tasks repeat broad verbs instead of naming the states to implement at `/Users/mateicanavra/Documents/.nosync/DEV/worktrees/wt-codex-deep-habitat-openspec-remediation/openspec/changes/deep-habitat-d5-baseline-authority/tasks.md:14`.

This blocks acceptance because D7 and D8 depend on D5 as a contract. A later execution agent could satisfy the two current scenarios while leaving multiple source-packet states ambiguous or unchanged.

### P1-2: D5/D8 ownership is ambiguous and can create a dependency cycle or authority leak

The packet index says D5 requires D0 and D2 and enables D7/D8. The source D5 packet says D5 adds a baseline contract projection consumed by Pattern Governance, while Pattern Governance must not bypass baseline contracts. The generated D5 artifacts instead say D5 will "Connect baselines to D2 registry facets and D8 governance admission" at `/Users/mateicanavra/Documents/.nosync/DEV/worktrees/wt-codex-deep-habitat-openspec-remediation/openspec/changes/deep-habitat-d5-baseline-authority/proposal.md:27` and `/Users/mateicanavra/Documents/.nosync/DEV/worktrees/wt-codex-deep-habitat-openspec-remediation/openspec/changes/deep-habitat-d5-baseline-authority/design.md:24`.

That phrasing has two unsafe interpretations: D5 edits or defines D8 admission behavior, which violates Pattern Governance ownership, or D5 depends on D8 admission being defined, which contradicts the D5 -> D8 ordering. The intended contract should be one-way: D5 owns baseline debt states and exposes the minimal baseline projection/refusal result that D8 later consumes. D8 owns pattern lifecycle/admission.

This blocks acceptance because the packet currently leaves two possible owners for "baseline relation to governance admission" and does not state the consumer projection D8 receives.

Add design-time validation gates separately from later implementation gates so the packet can say which commands were run now (`openspec validate`) and which commands must be run after source edits.

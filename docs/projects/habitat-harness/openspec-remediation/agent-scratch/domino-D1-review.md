## Acceptance Verdict

Not accepted. D1 remains blocked as a draft scaffold.

The packet has the right instinct: it challenges inherited `Proof*` language and re-centers on command receipts, checks, diagnostics, transactions, refusals, and handoff records. But it does not yet perform the packet-specific design work it claims to perform. The remediation frame requires each packet to be executable without asking an implementation agent to reopen product, domain, naming, sequencing, or trade-off decisions (`openspec-remediation-frame.md:12-15`) and says not to advance a domino when terminology is inherited, public JSON compatibility is undecided, implementation must decide product/domain trade-offs, or validation gates lack expected results and non-claims (`openspec-remediation-frame.md:139-152`). D1 currently violates all four.

## P1 Findings

1. P1 - The D1 scaffold does not include the required schema/non-claim/current-test inventory, so the target receipt contract is not actually resolved.

   The source D1 packet requires design proof before implementation: schema inventory for existing proof shapes, non-claim inventory, and current tests that assert proof output (`D1-proof-contract-boundary.md:97-103`). The generated scaffold replaces that with broad implementation tasks: "Replace target-domain proof vocabulary..." and "Define bounded DTO contracts..." (`tasks.md:14-16`). Those tasks are still design questions. Current code shows multiple concrete surfaces that need row-by-row disposition: `AdapterProofArtifact`, `ProofArtifactWriter`, `proofClass`, `proofId`, and `nonClaims` in `proof-artifact.ts:7-42`; `VerifyProof` in `command-engine.ts:101-167`; `HookTrace` and hook proof output in `hooks.ts:119-154`; and Grit apply transaction proof in `grit-apply.ts`. Without a packet-local inventory, an implementer must decide which surfaces are receipts, compatibility wrappers, command-only DTOs, internal traces, or downstream-owned artifacts while coding.

2. P1 - Public-surface compatibility is circular because D1 requires D0 output that is not present in the D1 packet and is still pending.

   D1 requires "D0 command surface inventory" (`proposal.md:37-39`) and says implementation must have D0 compatibility disposition for every public command, JSON, export, script, target, generator, and hook surface touched (`design.md:47-50`). But the D0 scaffold itself is still draft and its downstream ledger says the D0 compatibility matrix is pending. D1's own downstream ledger also says "D0 compatibility matrix | pending | Confirm public surfaces touched by D1 are classified before implementation" (`downstream-realignment-ledger.md:5-6`). That leaves the public-surface decision outside the accepted packet. D1 cannot be accepted as an implementation-control packet until either the exact D0 rows it consumes are attached/cited, or D1 is explicitly marked blocked behind an accepted D0 compatibility matrix.

3. P1 - The domain boundary is too broad and the owner language hides multiple authorities.

   The scaffold names a single owner, "Command Receipt Contract" (`design.md:15-20`), but then assigns it verify, check, hook, Grit adapter, apply transaction, and compatibility artifact outputs (`design.md:24-26`). The original packet deliberately separated forbidden owners: check must not define proof semantics beyond diagnostics, hooks must not define proof semantics beyond local feedback, apply must not define current-tree diagnostic proof, and Graphite state must not collapse into command proof (`D1-proof-contract-boundary.md:19-24`). The new scaffold does not preserve that authority map in target terms. It risks recreating a shared receipt/proof super-domain that every later packet can claim. This fails the domain-design authority test: one named owner is not enough when several subcontracts have different users, failure modes, and adjacent owners.

## P2 Findings

1. P2 - The OpenSpec spec delta is too thin to prevent impossible states or lazy terminology.

   The spec has one generic SHALL and two scenarios (`spec.md:3-13`). It does not define the allowed receipt/diagnostic/transaction/refusal state families, required fields, non-claim vocabulary, status derivation, failure/refusal projection, bounded stream policy, or compatibility wrapper rules. The scenario "Legacy proof-shaped DTO remains public" says D1 either preserves or version-renames a `Proof*` DTO (`spec.md:11-13`), but does not choose which. That is the exact decision the packet is supposed to make before implementation.

2. P2 - Validation gates are named as commands, not falsifying gates with expected results, bad cases, cache stance, and non-claims.

   The remediation frame says validation gates must name exact commands, expected results, and non-claims (`openspec-remediation-frame.md:151`). D1's proposal and tasks list commands only (`proposal.md:75-81`, `tasks.md:18-24`). The source packet had stricter expectations: expected exit 0 for schema tests, `habitat check --json` status conditions, `git status --short --branch`, cache stance, one malformed proof payload, one command failure projection, and a non-claim that no individual rule is proved (`D1-proof-contract-boundary.md:136-148`). The scaffold drops or weakens those oracles. It also omits `habitat verify --json`, even though the main current public shape is `VerifyProof` (`command-engine.ts:101-167`) and the verify command advertises JSON mode as a structured `VerifyProof` artifact.

3. P2 - The implementation write set and protected paths are absent from the packet.

   The proposal says the expected Habitat implementation write set is named in `design.md` (`proposal.md:49-54`), but `design.md` only says the executor must have a concrete write set before implementation (`design.md:47-52`). `tasks.md` repeats that the phase record should record the write set later (`tasks.md:7-10`). This is not an executable OpenSpec packet. At minimum the packet must name candidate source files, tests, docs, generated/read-only exclusions, and forbidden adjacent packet paths.

4. P2 - Downstream effects are underspecified and sequencing diverges from the source packet without rationale.

   The source packet says D1 unblocks D6, D7, D8, D9, D10, D11, D12, D13, and D14 (`D1-proof-contract-boundary.md:43-50`). The generated proposal enables only D6, D7, D9, D11, and D12 (`proposal.md:41-47`). The packet index follows that narrower dependency set, but the D1 design does not explain why D8, D10, D13, and D14 no longer need receipt/proof language. The downstream ledger is generic pending work (`downstream-realignment-ledger.md:5-9`) rather than a per-domino assumption table. That ambiguity can let later packets either ignore D1 when they should consume it, or reinvent proof/receipt language locally.

5. P2 - Historical proof/artifact language in durable docs and user-facing hook output is not dispositioned.

   The remediation frame permits historical docs to retain proof/evidence prose when operationally clear, but treats proof/evidence-shaped product code, types, and target language as suspect (`openspec-remediation-frame.md:55-79`). Current durable docs and hook output still expose proof wording, including "hook proof: local feedback only; CI remains authoritative" (`hooks.ts:152-154`, emitted at `hooks.ts:210-214` and `hooks.ts:403-405`), "structured proof artifact" in `SCENARIOS.md`, and "Proof Coverage" / "proof artifacts" in `IMPLEMENTED-SURFACE.md`. D1 says to update docs/examples only when implementation facts change (`downstream-realignment-ledger.md:7`), but does not classify which occurrences are target product language, compatibility text, or acceptable historical/current-state prose.

  `| Current Surface | Current Path | Public Classification | Target Contract | Compatibility Decision | Owner | Forbidden Owners | Required Tests | Non-Claims | Downstream Consumers |`

- Replace "Define bounded DTO contracts for verify, check, hook, Grit adapter, apply transaction, and compatibility artifact outputs" with concrete rows, for example:

  - `CheckReport` remains the command JSON check result for `habitat check --json`; `ok` must be derived from rule statuses by constructor/validator; it is not a handoff receipt and must not claim Nx, Graphite, apply, runtime, or product completion.
  - `VerifyProof` is either preserved as a schemaVersion 1 compatibility wrapper or renamed/versioned to `VerifyHandoffReceipt`; D1 must choose after D0 classification and must test both JSON and human-output truth.
  - `HookTrace` remains local feedback telemetry; hook output must say local feedback only without claiming CI/review proof.
  - `GritApplyTransactionProof` is either renamed to an apply transaction record or explicitly retained as compatibility language with a scenario justification.
  - `AdapterProofArtifact` must be dispositioned against its hard-coded older path under `openspec/changes/habitat-effect-grit-adapter/workstream/proofs` (`proof-artifact.ts:127-139`): preserve, move, rename, or remove.

- Expand `spec.md` into separate requirements:

  - Requirement: Check results cannot claim receipt semantics beyond structural diagnostics.
  - Requirement: Verify handoff receipts bound check, Nx affected, streams, post-state, and non-claims.
  - Requirement: Hook traces are local feedback only.
  - Requirement: Apply transaction records distinguish dry-run, live apply, rollback, formatter handoff, and failed gates.
  - Requirement: Legacy proof-shaped public DTOs require D0-backed preserve/version/rename decisions.

- Replace validation bullets with:

  `Run <command>; expected <status>; proves <specific contract>; does not prove <non-claims>; cache stance <fresh/cache allowed/recorded>; injected bad case <name>.`

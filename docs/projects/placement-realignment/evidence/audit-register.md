# LANE stage-anatomy

SUMMARY: Placement (9 steps, mods/mod-swooper-maps/src/recipes/standard/stages/placement/) has clean per-product step boundaries with effect-tag ordering (wonders → surface-prep → resources → starts → discoveries → advanced-starts → terminal evidence step), but diverges structurally from foundation/morphology in five ways. (1) It is one monolithic stage mixing four product sub-domains, a projection-phase step (plot-landmass-regions, phase \"gameplay\"), and engine maintenance, while morphology/hydrology/ecology were split into focused sub-stages. (2) Major decision logic (resource type reassignment, start selection/spacing, hemisphere slotting) lives in 600–1100-line recipe-layer materialize files, not domain ops; foundation steps are thin op-callers. (3) Planning consumes undeclared engine readbacks (per-tile terrain/biome/feature; globalThis.GameInfo) bypassing artifacts. (4) Zero artifact validators vs validators everywhere else; plans double-published as overlapping artifacts with cross-step helper imports. (5) Empty knobs schema and hand-authored shadow public schemas vs op-derived ones. Dead weight: orphaned stages/resources, unwired domain/resources ops, permanently-zero output fields.

KEY FACTS:
- Placement stage executes 9 steps in declared order: derive-placement-inputs, plot-landmass-regions, place-natural-wonders, prepare-placement-surface, place-resources, assign-starts, place-discoveries, assign-advanced-starts, placement (mods/mod-swooper-maps/src/recipes/standard/stages/placement/index.ts:29-39).
- Natural wonders are stamped BEFORE engine maintenance (validateAndFixTerrain/recalculateAreas/storeWaterData) by design; prepare-placement-surface requires the naturalWondersPlaced tag (prepare-placement-surface/contract.ts:19-23).
- plot-landmass-regions is the only placement step with phase "gameplay" (projection phase) and publishes mapArtifacts.projectionMeta; all siblings are phase "placement" (plot-landmass-regions/contract.ts:9-14).
- place-resources/materialize.ts is 1134 lines and re-decides resource type per plot at materialize time (assignResourceIntents), overriding the domain planResources output (materialize.ts:400,1073).
- derive-placement-inputs reads live engine terrain/biome/feature per tile into wonder planning (derive-placement-inputs/inputs.ts:75-93,193-195) despite the contract doc claiming readback drift is avoided (contract.ts:10-16).
- readRuntimeResourceCatalog reads globalThis.GameInfo.Resources directly in the recipe layer (place-resources/materialize.ts:772-793).
- Foundation and morphology stage code contains zero adapter.* calls; placement steps mutate the engine directly (setStartPosition, placeResourceIntent, setLandmassRegionId, recalculateFertility, assignAdvancedStartRegions).
- Placement implements all artifacts with empty {} (no validate hooks); foundation/morphology/hydrology/ecology stages all register validators (grep validate: hits 0 in placement, 10 files in foundation).
- PlacementKnobsSchema is empty; placement public schemas are hand-authored shadows of op configs rather than derived like foundation's defaultStrategyConfigSchema (placement-public-config.ts:7-14 vs foundation/index.ts:19-37).
- stages/resources/ contains only artifacts.ts with zero imports anywhere in src; domain/resources planning ops (aquatic/cultivated/geological/terrestrial/groups) are wired only into tests, not the recipe.
- placementOutputs.floodplainsCount and snowTilesCount are hardcoded 0 at the only publish site (placement/apply.ts:174-175) yet validated >=0 by tag satisfaction logic (tags.ts:294-301); methodCalls is never produced.
- The terminal 'placement' step performs no engine mutations: it verifies product artifacts, snapshots engine heightfield, computes waterDriftCount, and publishes placementOutputs/engineState/placementEngineTerrainSnapshot (placement/apply.ts:60-180).
FINDINGS:
## [P2/high] Placement is a single monolithic stage spanning four product sub-domains plus projection and engine maintenance
While morphology was split into 4 focused sub-stages (+map-morphology), hydrology into 3, and ecology into 3, placement remains one 9-step stage covering wonders, resources, starts, discoveries, advanced starts, engine maintenance (prepare-placement-surface), a projection step (plot-landmass-regions, declared phase "gameplay" while all sibling steps are phase "placement"), and a terminal evidence-collation step confusingly named "placement" inside stage "placement". Product config for all sub-domains is front-loaded into derive-placement-inputs rather than per-product steps. Step boundaries are correct, but the stage boundary diverges from the split-stage pattern the better-regarded stages established.

# LANE player-starts

SUMMARY: Starts are computed entirely script-side: a plan-starts op scores per-tile viability (land envelope, pedology fertility, moisture/aridity/temperature, river/lake freshwater, placed-resource support, elevation roughness) and a materializer picks tiles per west/east slot and writes them via adapter.setStartPosition mid-stage. The base-game engine positioner (assignStartPositions/chooseStartSectors) is wrapped in the adapter but never called; the only engine-side start work that runs is assignAdvancedStartRegions (a different concept: advanced-start regions) after discoveries. The "two layers" impression is real but one layer is dead code plus inert sector machinery: startSectors is always [] and the public config force-overrides it to [], so all sector knobs/filters/viz no-op. Major gaps vs base game: no mountain/impassable/natural-wonder exclusion, no civ/leader start biases, no alive-major player-id mapping (positional index used as playerId, counts from mapInfo defaults), and a desperation fallback that ignores viability entirely.

KEY FACTS:
- Script-side planner is the sole start authority; adapter wrappers for base-game assignStartPositions (packages/civ7-adapter/src/civ7-adapter.ts:1080) and chooseStartSectors (:1114) have zero callers in mod/recipe code.
- Starts take engine effect inside the assign-starts step via adapter.setStartPosition per player (mods/mod-swooper-maps/src/recipes/standard/stages/placement/steps/assign-starts/materialize.ts:157,182,207), after resources and before discoveries (stages/placement/index.ts:29-39).
- assign-advanced-starts is a separate engine-side pass (recalculateFertility + assignAdvancedStartRegions) gated on discoveriesPlaced (steps/assign-advanced-starts/index.ts:18-24); it assigns advanced-start REGIONS, not start plots.
- startSectors is always [] (recipes/standard/runtime.ts:38,73) and placement-public-config.ts:251 force-overrides overrides.startSectors=[], so sector filtering in materialize.ts:422-453 always returns all candidates.
- startSectorRows/Cols default to mapInfo ?? 4 (runtime.ts:36-37) and are threaded through op contract/output (domain/placement/ops/plan-starts/contract.ts:11-20) despite having no effect.
- Scoring inputs wired in steps/assign-starts/index.ts:45-62: topography.landMask/elevation, landmasses, coastlineMetrics, hydrography.riverClass, lakePlan.lakeMask, biomeClassification effectiveMoisture/surfaceTemperature/aridityIndex, pedology.fertility, placed resource indices.
- Candidate filter requires only land + non-lake + region slot (domain/placement/ops/plan-starts/strategies/default.ts:361-379); morphology mountainMask exists (stages/morphology/artifacts.ts:149) but is not consumed, so mountain/natural-wonder tiles are eligible starts.
- Base game excludes water and mountains and uses engine StartPositioner.getStartPositionScore plus per-civ start bias (.civ7/outputs/resources/Base/modules/base-standard/maps/assign-starting-plots.js:923-930, adjustScoreByStartBias).
- Base game maps start slots to Players.getAliveMajorIds() and handles humans/hemispheres (assign-starting-plots.js:218-243); the mod uses positional index i as playerId and mapInfo default counts (?? 4) regardless of alive players (runtime.ts:34-35).
- Selection ranks 0.86*viability + 0.14*spacing per tier, relaxing spacing one tile at a time until candidates exist (materialize.ts:491-527); fallbacks: openPool ignores region slots (:165-188), desperation picks any slotted land tile by max distance with no viability (:190-212,546-576).
- Twelve public start knobs exist in placement-public-config.ts:100-220 mirroring the strategy schema (contract.ts:186-270); none control sector layout, player counts, coastal bias, or start biases.
- Hard failure if any player cannot be seated even after desperation (materialize.ts:214-218); final placement step only validates startsAssigned >= expectedPlayers (recipes/standard/tags.ts:309-310).

# LANE resources

SUMMARY: Live resource placement is a two-phase pipeline: placement/plan-resources scores land tiles with one generic suitability scalar and greedily selects density*landTiles/100 plots with global min-spacing (default.ts:113-165); place-resources/materialize.ts then re-derives everything via a three-phase least-used-type assignment over engine-legal plots (materialize.ts:400-567), discarding the plan's per-plot preferred types and force-equalizing per-type counts to floor(target/numTypes). Public knobs are only densityPer100Tiles, minSpacingTiles, maxPlacementsPerResourceShare. Per-resource sparsity, resource-resource affinity/exclusion, near-start guarantees, and regional/hemisphere equity are inexpressible. Civ7 policy constrains placement via age-gated candidate filtering (initial-map authoring policy) and live ResourceBuilder.canHaveResource legality; the planner cannot see biome legality, so materialization fights it via whole-map fallback. Starts are assigned after resources and score nearby placed resources (one-directional). A richer per-resource model (family ops, expectedCountRange, habitat lanes) exists in domain/resources but is wired only into tests.

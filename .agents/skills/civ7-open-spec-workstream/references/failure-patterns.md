# Failure Patterns

| Failure | Symptom | Repair |
|---|---|---|
| Unaccepted spec becomes authority | Phase docs soften or replace controlling product/architecture authority | Restore authority order; accepted baselines can control a workstream, ordinary phase notes cannot |
| Proposal-only theater | Phase artifacts exist but code, gates, and downstream updates never follow | Convert to a full phase loop |
| Code-first drift | Implementation starts before owners, write set, and gates are explicit | Stop coding and repair phase artifacts |
| Review theater | Agents review but findings do not change artifacts, code, tests, or closure claims | Disposition every finding and patch control surfaces |
| Shortcut language survives | Fallback, temporary, compatibility, optional, or dual path is authorized by ambiguity | Remove it or record explicit authority |
| Current-code preservation | Tasks protect stale wrappers, barrels, stage ids, generated outputs, or hidden sub-pipelines | Return to target authority and preserve guarantees, not containers |
| Downstream drift | Later docs/issues/specs assume pre-phase surfaces | Run realignment before closure |
| Compaction amnesia | The next agent needs chat history to know what is true | Update phase record and Next Packet |
| Dirty closure | Phase ends with unexplained dirty files or untracked artifacts | Classify, commit, clean, or hand off owned dirty state |

## Black-Ice Checks

- Are any tasks still design questions?
- Are two owners possible for one concern?
- Is a generated artifact being treated as source?
- Is a reviewer finding acknowledged only in prose?
- Is downstream work assuming a surface this phase changes?
- Does closure claim more than verification proves?

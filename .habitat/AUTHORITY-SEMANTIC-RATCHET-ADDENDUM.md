# Semantic Ratchet Architecture

Status: `draft-normative-addendum`
Owner: Habitat authority stewardship
Scope: Process strategy and work-container geometry for ratcheting architecture
authority into executable repository habit

This document defines the general process vocabulary behind future Habitat-led
architecture work. It is a normative general-method addendum to the Habitat
authority model: it names the recursive work containers, state changes, proof
boundaries, and drift posture by which intuition becomes durable system
authority. It does not compete with or replace
`.habitat/.active/frames/BLUEPRINT-AUTHORITY-RATCHET-DESCENT-FRAME.md`.
That active frame is the controlling specialization whenever the selected work
is a blueprint-authority descent.

The phrase "semantic pressure chamber" is useful shorthand for the inner
mechanism, but the normative process name is:

```text
Semantic Ratchet Architecture
```

The point is not to think clearly once. The point is to make the system unable
to fully forget what it learned.

## General Method And Active Specialization

This addendum generalizes semantic-ratchet work across product architecture,
ownership, runtime realization, projection, and other Habitat-led
transformations. The active blueprint descent frame specializes that general
method for one constructible kind at one selected depth. When both apply, the
descent frame controls its container geometry, row obligations, proof classes,
ratchet closure, and ascent conditions; this addendum supplies the broader
semantic vocabulary and does not resequence those gates.

| General semantic-ratchet concept | Blueprint-descent specialization |
| --- | --- |
| Field and frame | Current boundary, future boundary, authority owner, exterior, and reframe trigger. |
| Packet and workstream | Selected blueprint kind, selected depth, red corpus, survivor law, and opened workstream. |
| Slice and domino | Row-ledger execution units that reduce the selected red corpus without losing obligations. |
| Proof | Separate Record truth, Native tool behavior, Habitat wrapper behavior, runtime/product proof, review disposition, and workflow closure claims. |
| Guardrail | Positive kind law plus retained supporting rails after duplicate or absorbed rules are removed. |
| Overwatch | Post-ratchet revalidation, rule-surface cleanup, ascent, and explicit next-container selection. |

The specialization may expose a reusable principle. Habitat stewardship should
promote that principle into this addendum only when it holds outside one
blueprint kind or descent. Instance-specific packet names, selected depths,
branch state, and rule ledgers remain in the active frame or workstream.

## Core Thesis

Semantic ratchet work is a recursive architecture-stewardship process that
turns intuition into durable system authority through pressure, proof, and
ratchet.

The inner motion:

```text
intuition
  -> vocabulary
  -> frame
  -> authority model
  -> spec
  -> work containers
  -> proof gates
  -> guardrails
  -> system habit
```

The semantic pressure chamber is the compression step. A blurry but important
idea is put under pressure from multiple sides until it stops being vibes and
becomes a usable operating distinction.

## What Makes This Different

Normal planning asks:

```text
What needs doing?
Who does it?
When is it done?
```

Semantic ratchet work asks:

```text
What kind of thing is this?
What owns its truth?
What projects it?
What can observe it?
What would prove it?
What prevents drift after it lands?
```

The work containers are not merely task containers. They are meaning
containers. Each preserves a semantic commitment at a different altitude.

## Container Stack

### Field

The broad domain of concern.

Example:

```text
Civ7 modding as a foundry problem
```

The field is not yet the frame. It is the space from which relevant signals are
selected.

### Frame

The chosen lens, boundary, and salience model.

Example:

```text
services own truth, plugins project, apps select
```

A frame selects what is in, what is foregrounded, and what is exterior. A frame
must name what would force a reframe.

### Packet

A durable semantic artifact that can travel across agents, compactions,
branches, and workstreams.

Examples:

```text
future Civ7 foundry architecture reference
MapGen architecture normalization packet
semantic ratchet addendum
```

A packet should carry enough authority and context that future work can begin
from it rather than rediscovering the frame from transcript history.

### Workstream

A governed transformation lane.

Example:

```text
MapGen as pilot service
```

A workstream owns a bounded architectural transformation. It does not become
the entire product, the entire repository, or the whole future architecture.

### Slice

One executable architectural move.

Example:

```text
classify current packages by authority
```

A slice is complete when it moves one concrete, testable piece of the target
architecture forward and leaves proof behind.

### Domino

The smallest change that advances the next harder change.

Example:

```text
make runtime resources explicit before moving folders
```

A domino is not just a small task. It is a change chosen because it reduces the
uncertainty or friction of the next architectural move.

### Proof

Evidence that a claim is true at its boundary.

Examples:

```text
doc proof
type proof
Habitat check proof
generated artifact proof
runtime readback proof
in-game evidence
```

Proof must be scoped. A generated artifact proves generation happened. It does
not define product policy. A TypeScript check proves a local type boundary. It
does not prove in-game correctness.

### Guardrail

The ratchet that prevents regression after proof exists.

Examples:

```text
import rule
Habitat check
doc routing rule
generated-output read-only rule
public contract gate
```

Guardrails should encode achieved structure. They should not red-bar the repo
before the migration that makes them pass.

### Overwatch

Cross-cutting drift detection after the system moves.

Examples:

```text
authority drift
router drift
proof drift
generated-artifact drift
projection/truth drift
```

Ownership is local accountability. Overwatch is cross-cutting attention: it
looks for cascades, ratchets, boundary erosion, duplicated patterns, stale
docs, and hidden coupling.

## Process Loop

The process is recursive:

```text
1. Compress the latent idea into language.
2. Stress the language against current reality.
3. Split truth from projection, source from artifact, proof from claim.
4. Name the owning authority.
5. Create a container that can carry the distinction.
6. Execute one slice that makes the distinction real.
7. Prove it.
8. Ratchet it into tooling, docs, or workflow.
9. Watch for drift.
```

Each loop should either clarify authority, enable execution, close proof, or
prevent drift. If it does none of those, it is decorative and should be cut.

## State Model

Semantic ratchet work moves through state changes rather than checklist items.

### Latent Signal

An important pattern exists but has not yet been named well enough to govern
work.

Valid work:

- gather evidence;
- name tensions;
- test vocabulary;
- identify false owners and mixed concepts.

Invalid work:

- create guardrails;
- move source by vibes;
- turn early language into policy.

### Frame Pressure

The idea is compressed into candidate distinctions and tested against reality.

Valid work:

- generate structural alternatives;
- name what is in, foregrounded, and exterior;
- test against current repo behavior;
- find falsifiers.

Invalid work:

- accept the first plausible vocabulary;
- preserve current package names as architecture;
- confuse recognition with construction.

### Authority Commitment

The distinction has an owner, scope, and claim strength.

Valid work:

- state which authority owns truth;
- name forbidden owners;
- classify proof boundaries;
- record what remains draft, candidate, or normative.

Invalid work:

- leave ownership implicit;
- encode unresolved questions as fallback behavior;
- let generated output become policy.

### Containerization

The commitment is placed in the right durable work container.

Valid work:

- choose frame, packet, workstream, slice, domino, proof, guardrail, or
  overwatch;
- keep temporal context out of evergreen authority unless promoted;
- keep standalone packets self-sufficient.

Invalid work:

- create a document because the idea feels important;
- bury durable authority in chat logs;
- put project status in canonical docs.

### Execution

One slice makes the commitment real in source, docs, tests, or tooling.

Valid work:

- move by authority, not current paths;
- implement only after destinations are deterministic;
- keep one story testable end to end.

Invalid work:

- broad folder reshuffles without an ownership map;
- new shared buckets;
- compatibility shims that hide the old mixed authority.

### Proof Closure

The slice is verified at the right boundary.

Valid work:

- run the nearest checks;
- label proof honestly;
- distinguish local, generated, runtime, and in-game evidence.

Invalid work:

- generalize one proof class into another;
- claim runtime correctness from static checks;
- treat review disposition as behavioral proof.

### Ratchet

The system is made less able to regress.

Valid work:

- add guardrails after the target shape exists;
- update routing docs;
- remove absorbed duplicate rules;
- tighten Habitat authority.

Invalid work:

- guard future intent before implementation exists;
- keep negative sentries after positive law absorbs them;
- let temporary scaffolding become architecture.

### Overwatch

The system watches for drift after the move.

Valid work:

- identify named cascades;
- track stale docs or repeated repair patterns;
- schedule follow-up where explicit triggers exist.

Invalid work:

- declare the architecture done;
- leave hidden coupling invisible;
- rely on memory of why a move happened.

## Container Selection Rules

Use a **frame** when the lens or boundary is not yet stable.

Use a **packet** when the frame must travel and future work should not
rediscover it.

Use a **workstream** when a transformation spans multiple slices and needs
governed sequencing.

Use a **slice** when one bounded architectural move can be implemented and
verified.

Use a **domino** when the value of the change is primarily that it unlocks the
next harder move.

Use **proof** when a claim must be closed at a specific boundary.

Use a **guardrail** only after the structure being guarded exists or the
guardrail is explicitly a candidate/non-enforcing draft.

Use **overwatch** when the risk is slow drift, hidden coupling, or regression
through later unrelated work.

## Failure Modes

### Architecture Theater

Symptom:

```text
beautiful containers that do not reduce entropy
```

Correction:

```text
cut any container that does not clarify authority, enable execution,
close proof, or prevent drift
```

### Premature Guardrail

Symptom:

```text
a rule exists before the target shape can pass
```

Correction:

```text
demote to candidate, migrate first, then ratchet
```

### Package-Shape Capture

Symptom:

```text
current package names are treated as semantic owners
```

Correction:

```text
classify by truth, projection, selection, resource, provider,
realization, and observation
```

### Proof Inflation

Symptom:

```text
one evidence class is used to support claims outside its boundary
```

Correction:

```text
rename the proof claim or run the missing proof
```

### Work-Management Substrate Drift

Symptom:

```text
the system becomes excellent at managing work but weak at completing product
change
```

Correction:

```text
re-anchor every container to product gravity and executable proof
```

## Relationship To Future Civ7 Foundry Architecture

`FUTURE-CIV7-FOUNDRY-ARCHITECTURE.md` is the target product/technical
architecture reference.

This addendum is the method by which Habitat should ratchet that architecture
into the repo:

```text
future architecture
  -> ownership map
  -> pilot service
  -> explicit resources/providers
  -> projection surfaces
  -> runtime realization
  -> observation
  -> guardrails
  -> overwatch
```

The future architecture should not be implemented by mass folder rename.
Habitat should first make authority explicit, then descend through bounded
slices that prove and ratchet one layer at a time.

For a blueprint-kind slice, "descend" has the exact meaning and closure gates
defined by
`.habitat/.active/frames/BLUEPRINT-AUTHORITY-RATCHET-DESCENT-FRAME.md`.
For a non-blueprint architecture slice, this addendum remains the general
container vocabulary, and the opened workstream must supply an equally explicit
specialization before execution.

## Final Rule

Semantic ratchet work is justified only when it changes the system's future
behavior.

The output of a good ratchet is not a better explanation. The output is a
repository that can act differently because the explanation became authority,
proof, and habit.

## WHAT

This frame treats the post-Domino-39 problem as a position decision, not a
default march into artifact gathering. Dependency-tag clarified edge/tag
authority and rejected false siblings, but it did not by itself prove a coherent
artifact input set. The next move should select the largest bounded domino that
reduces false destination pressure: artifact gathering only if whole-rule
artifact-value authority is visible, otherwise targeted garbage collection or
candidate `_blueprints` pruning.

## WHY

The session's last turn correctly moved away from projection-contract churn and
into destination simplification. Domino 39 then produced a small but important
signal: one rule moved cleanly into `dependency-tag`, while adjacent
effect-tag, tag-catalog, projection-contract, config-surface, and
artifact-contract labels were rejected as sibling blueprints. If we now assume
artifact is automatically next, we risk recreating the same over-splitting in a
new vocabulary. If we instead run broad classification, we recreate hidden
taxonomy. The stronger frame is to use Domino 39 as a discriminator: changed
tree first, next largest bounded slice second.

## Construction History

Structural alternative considered: artifact blueprint gathering immediately
after dependency-tag.

Why rejected or demoted: artifact is real, but the visible residue is still
mixed across map-output, morphology/story overlay, config, Studio artifact
supply, generated resource policy, visualization build artifacts, and manifest
metadata. A real artifact slice must start from artifact value/schema/publish/read
whole predicates, not every rule containing the word artifact.

Structural alternative considered: repo-wide garbage cleanup now.

Why rejected or demoted: garbage pressure is real, especially in foundation,
morphology, config, cutover, and runtime remainders, but cleanup should retire,
replace, or consolidate concrete rows. A generic cleanup branch would flatten
the same dependency order the destination frame was built to preserve.

Structural alternative considered: prune all `_blueprints` now.

Why rejected or demoted: false candidate blueprints remain a major source of
confusion, but pruning all of them globally would become corpus classification.
The useful version is targeted pruning of the candidate family that most
distorts the next real move.

## Hard Core And Protective Belt

Hard core:

1. Authority Activation remains the objective: admitted kinds, honest niches,
   staged remainders, and execution projections must become legible without
   reverse-engineering packet history.
2. Dependency-tag is now real blueprint authority, but its small movement is a
   discriminator rather than a mandate that artifact must be next.
3. Artifact is a real blueprint kind only for artifact values/contracts:
   stable id, schema, publish/read behavior, immutable store semantics, and
   buffer exception. Artifact does not absorb dependency-tag governance,
   generated-output hygiene, or manifest `operation.kind` metadata.
4. The next branch must choose between artifact, garbage, or `_blueprints`
   pruning from current tree evidence, not from the previous session's
   momentum.
5. Fake sibling destinations are defects in the search space. Merging,
   demoting, or deleting them is progress when it makes the next real slice
   more mechanical.

Protective belt:

- Artifact gathering may still be next if a tight candidate set of
  artifact-value/schema/publish/read rows appears after a focused re-read.
- Config may later become a niche or public-config owner, but not a blueprint
  unless a bounded slice proves constructibility.
- Platform, Studio, docs, toolkit, resources, and visualization candidates may
  become real authority later; each must pass constructibility and whole-rule
  fit instead of inheriting authority from `_blueprints` directories.
- Repo-wide hygiene should be handled soon as a separate branch if it blocks
  trust in closure gates.

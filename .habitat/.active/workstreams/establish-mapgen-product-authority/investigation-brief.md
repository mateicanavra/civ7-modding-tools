- Why this geometry: the source set is scattered and mixed between evergreen docs, project baselines, specs, and source. Bottom-up single-rule analysis is the failure mode being corrected.

## 6. Evidence Policy

- Evidence standard: source-backed, provenance-bearing claims; no unsourced summary as authority.
- Source authority order:
  1. Current user decision and explicit repo instructions.
  2. Root/subtree `AGENTS.md` and repo process docs.
  3. Accepted project baselines, especially `docs/projects/engine-refactor-v1/architecture-normalization-packet.md`.
  4. Canonical evergreen docs such as `docs/PRODUCT.md`, `docs/SYSTEM.md`, `docs/PROCESS.md`, `docs/system/ARCHITECTURE.md`, and `docs/system/TESTING.md`.
  5. Canonical MapGen/Swooper Maps/SDK docs under `docs/system/**`.
  6. ADRs and durable deferrals.
  7. Active project notes after classifying status.
  8. Current source and tests as implementation/constructibility evidence.
  9. Official resources as game-data evidence.
  10. Generated output as proof of generation only.
  11. OpenSpec/change records as downstream change-management unless promoted.
  12. Archived docs/chat summaries as discovery material only.
- Minimum corroboration: one authority source plus source constructibility evidence for accepted claims when source shape is relevant; conflicts recorded rather than averaged.
- Conflict rule: higher authority wins unless the source/code contradiction proves the authority is not constructible, in which case record the contradiction and required decision.
- Claim-strength labels allowed: accepted authority; intended authority not fully built; source constructibility evidence; source contradiction; stale/superseded; discovery lead; open decision.
- Required uncertainty markers: owner unknown; name collision risk; source-only; stale-source risk; missing constructibility evidence.
- What does not count as evidence: unproven current folder shape, generated output as editable source, archived docs without re-grounding, package tests as junk drawers for stale policy, unsourced summaries.
- Code-vs-doc rule: code can prove current behavior, constructibility, or contradiction; docs/project baselines define intended authority unless explicitly stale or infeasible.

## 7. Rail Decision

- Selected rail: multi-agent source/rule/code non-mutating investigation.
- Why this rail: the corpus is broad, source categories differ, and independent lanes reduce context loss while preserving one accountable synthesis owner.
- Rejected rails: implementation cascade; full OpenSpec workstream; one-agent opportunistic reading; global Layer 1 rerun.
- Rail-bias risks: over-reading stale project notes; treating source topology as authority; producing a project log instead of reference; creating duplicate progress ledgers.
- Constraints passed to the rail: no rule/source/test/generated mutations; provenance required; one artifact per lane; no duplicate operational matrices; final reference must be normative and useful for Layer 2.
- Adapter notes: lane agents may write only their assigned artifact file.

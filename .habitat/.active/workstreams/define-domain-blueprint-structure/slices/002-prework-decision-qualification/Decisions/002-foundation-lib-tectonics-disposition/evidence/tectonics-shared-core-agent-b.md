## Core comparison conclusion

Existing-core replacements:

- `NeighborhoodMesh` type semantics are already represented by core `DelaunayMesh` as a superset, but a minimal core mesh input type may be useful only if new mesh helpers are added.
- `clamp01` is replaceable with existing configured core helper semantics: `clampPct(value, 0, 1, 0)` or `clampFinite(value, 0, 1, 0)`.

New core API candidates:

- `clampByte` as a rounded unsigned-byte quantizer (`quantizeU8`).
- `clampInt8` as a rounded symmetric signed-byte quantizer if infinity saturation is retained (`quantizeI8Symmetric` or equivalent).
- `normalizeToInt8` as a unit-vector-to-signed-byte vector encoder, building on existing vector helpers but pinning the `1e-9` zero threshold.
- `computeMeanEdgeLen` as a generic periodic CSR mesh metric.
- `findNearestCell` as a generic periodic point-mesh nearest-cell search.
- `chooseDriftNeighbor` as a generic mesh neighbor selection by vector projection, renamed without drift vocabulary.

Not core:

- `deriveResetThreshold`; this remains provenance reset policy unless a new accepted decision generalizes it.

Package-surface blockers:

- No `package.json` export blocker for math, grid, or mesh candidates: `./lib/math`, `./lib/grid`, and `./lib/mesh` already exist.
- Root package exports are uneven: root exports `clamp01`, `clampInt`, `clampPct`, but not `clampU8`, `clampFinite`, `wrapDeltaPeriodic`, grid vector helpers, or mesh helpers. Future execution can use subpath imports without changing root exports.
- A new `./lib/vector` subpath would require package surface work; avoid it unless an architecture decision wants vector math separated from current `lib/grid/vector-field.ts`.
